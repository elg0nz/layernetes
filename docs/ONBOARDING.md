description: How a new user gets an account and ships their first agent on the production cluster.

**Decision: closed registration, admin-provisioned accounts.** Gitea
self-registration is disabled (`DISABLE_REGISTRATION: true`,
[`ll-infra/values.yaml`](../ll-infra/values.yaml)) — there is no public
signup page to gate, because it doesn't accept registrations regardless of
who reaches it. Only the platform admin creates accounts.

A new user's entire onboarding:

1. Admin runs `hack/provision-users.sh <username>` against the cluster,
   which creates the Gitea account via the admin API and prints the
   generated password. Hand that to the user out of band.
2. `pipx install llnate` (or clone + install), then
   `llnate init my-agent && cd my-agent && llnate login` — the PAT, age
   keypair, cloud repo, and CI wiring all provision lazily on that first
   login. Nothing else to hand out.

## Why closed instead of gated

An earlier draft of this decision considered leaving registration open and
gating just `/user/sign_up` behind a Cloudflare Access policy (GitHub login
required) — self-serve, but access-controlled at the edge. That's not the
current design: only the admin should be able to create users, full stop, so
`DISABLE_REGISTRATION: true` does that directly at the source rather than
relying on an edge policy in front of a page that would otherwise work. One
fewer moving part, and one fewer thing (a Cloudflare Access application) to
provision, keep in sync with the identity provider, or accidentally
misconfigure open.

| Surface | Auth story |
| --- | --- |
| `gitea.…/user/sign_up` | disabled server-side; the form doesn't accept submissions no matter who reaches it |
| `gitea.…` (everything else) | open | Gitea session / PAT / basic auth |
| `api.…` (ll-api) | open | bearer token (Gitea PAT); requires an admin-provisioned account |
| `*.agents.…` | open | public by design (MCP + HTTP for anyone) |

## Provisioning a user

```sh
GITEA_URL=https://gitea.wtp.io GITEA_ADMIN=layernetes-admin \
GITEA_ADMIN_PASSWORD=... hack/provision-users.sh alice bob
```

Prints one `username password` line per created user; existing users are
left untouched. See the script's header comment
([`hack/provision-users.sh`](../hack/provision-users.sh)) for details — the
admin credentials come from the `ll-gitea-admin` Secret
(`gitea.admin.existingSecret` in `values.yaml`).

Local development uses the same script and setting (`DISABLE_REGISTRATION:
true` in every values file) — there's no environment where self-registration
is on.

## Trade-offs and the upgrade path

- Every new user needs the admin to run `provision-users.sh` (or create the
  account by hand) — no fully self-serve path. Acceptable at the current
  scale; revisit if onboarding volume grows past what a manual step handles.
- Users still manage a Gitea password (generated by the script, not
  self-chosen). The planned tier-2 upgrade — `llnate login` doing a real
  OAuth authorization-code flow against Gitea, with Gitea federating to
  GitHub — removes passwords entirely and needs no changes to keys, CI
  tokens, or the operator. The login endpoint shapes were kept
  OAuth-compatible for exactly this. Whether that OAuth flow ever allows
  *self-serve* account creation (vs. just passwordless login for
  admin-provisioned accounts) is a separate decision from this document.
